## Правила

- Миграции лежат в `/migrations/`, embed через `//go:embed all:../../migrations` в
  `internal/storage/storage.go`.
- Имя файла: `NNNN_описание.sql`, нумерация по возрастанию, применяются в
  лексикографическом порядке (`0001_init.sql`, `0002_...sql`).
- Раннер сам отслеживает применённые миграции в таблице `schema_migrations` — не
  писать `IF NOT EXISTS` как замену версионированию, миграции должны быть
  последовательными и неизменяемыми после мержа.
- **Никогда не редактировать уже смерженную миграцию.** Если нужно поменять схему —
  новый файл с следующим номером, даже если изменение кажется тривиальным (опечатка
  в имени колонки и т.п.). История миграций — это история реальных состояний БД у
  пользователей, которые уже могли её накатить.
- Каждая новая таблица/колонка должна явно указывать, к какому контуру относится:
  runtime (собственные матчи пользователя, `matches`/`wave_snapshots`/`recommendations`)
  или офлайн-кэш API (`units_reference` и подобные). Не смешивать в одной таблице.

## Проверка перед коммитом миграции

1. Накатывается ли она на чистую БД (`storage.Open` с новым файлом)?
2. Накатывается ли она поверх БД с уже применёнными предыдущими миграциями?
3. Если добавляется NOT NULL колонка в существующую таблицу — что происходит с уже
   существующими строками (SQLite требует DEFAULT для ALTER TABLE ADD COLUMN NOT NULL)?
