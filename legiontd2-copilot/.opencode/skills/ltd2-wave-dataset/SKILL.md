## Что уже собрано

~3000 матчей, ~11899 записей игроков, через `cmd/dataset/main.go` и
`internal/dataset/collector.go`, источник — `/games?includeDetails=true` (см.
`ltd2-api-v2` skill за форматом `buildPerWave`/`leaksPerWave`).

## Прежде чем добавлять новый сбор данных

- Проверить остаток дневного лимита API (1000 запросов/день) — батч-джоб сбора не
  должен молча упираться в 429 посреди ночного прогона. Логировать оставшийся лимит.
- Не дублировать уже собранные матчи — использовать `_id` матча как идемпотентный ключ
  при записи в `data/dataset/`.
- Матчи старше года не покрываются `/games` — для расширения датасета за 2018–2021
  нужен отдельный проход по архиву `.bson` (см. `ltd2-api-v2`), это отдельная задача,
  не путать с обычным сбором.

## Перед обучением модели на этих данных

- **Нет golden-датасета для CV/OCR** (см. ТЗ, TEST3) — этот датасет НЕ заменяет его.
  Это датасет для офлайн-обучения Advisor'а на исторических решениях игроков, а не
  эталон для проверки точности распознавания экрана.
- Учитывать `gameElo` и `gameResult` при формировании обучающей выборки — расстановки
  из низкоэлошных матчей не должны учиться как "оптимальные" наравне с топовыми.
- `leaksPerWave` — единственный прямой сигнал "волна провалена". Отсутствие протечки
  не равно "оптимальная расстановка", только "достаточная". Не путать эти два
  понятия при формулировке целевой переменной модели.

## Явно вне скоупа этого этапа

Детекция позиций юнитов с экрана через CV (Phase 2.3 из ТЗ) — начинается только
после того, как офлайн-модель на этом датасете показала пользу. Не начинать
разметку скриншотов "на всякий случай" параллельно — это параллельная работа без
подтверждённой необходимости.
