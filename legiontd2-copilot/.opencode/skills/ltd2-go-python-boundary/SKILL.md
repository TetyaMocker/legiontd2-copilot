## Инвариант, который нельзя нарушать

Go-код **никогда** не работает с изображениями/скриншотами напрямую. Весь CV/OCR —
только в `services/perception` (Python). Если видишь в Go-коде импорт чего-то вроде
image-обработки, декодирования PNG/JPEG для анализа пикселей — это нарушение границы,
даже если формально это "просто передать байты дальше".

## Известное расхождение с ТЗ (требует решения, не тихого смирения)

ТЗ фиксирует **gRPC** (см. `proto/perception.proto`) как протокол между Go и Python.
По факту в кодовой базе используется **REST HTTP POST** (`internal/http/server.go`
принимает `/api/ingest` от Python) + отдельно WebSocket для broadcast клиентам.

Это не является нарушением красных линий (`ltd2-constraints`), но это архитектурный
дрейф, который ТЗ не отражает. При следующей заметной работе с этой границей:

1. Либо мигрировать реализацию на gRPC, как задокументировано в `proto/perception.proto`
   (предпочтительно — типобезопасность, меньше самодельного JSON-контракта);
2. Либо явно обновить ТЗ, зафиксировав REST+WebSocket как осознанное решение, и
   удалить/архивировать `proto/perception.proto`, чтобы не вводить в заблуждение.

Не оставлять оба варианта одновременно существующими без явного выбора — следующий
человек (или следующая сессия агента) не должен гадать, какой из двух контрактов
актуален.

## Правило при добавлении нового поля в контракт

Любое новое поле, которое Python должен передать Go (например, новая метрика
уверенности OCR) — добавляется **сначала** в `proto/perception.proto` (или в
эквивалентный REST-контракт, если решение из пункта выше — REST), **затем**
реализуется по обе стороны. Не добавлять поля молча только на одной стороне.

## Частота вызовов и нагрузка

Тик-луп сейчас — 2 секунды. Это ориентир, не жёсткая константа — при добавлении
новых полей в `ReadEconomy`/`ReadBattlefield` пересчитывай, не превышает ли суммарная
нагрузка бюджет CPU из ТЗ (NFR2: не более 15–20% одного ядра в фоне).
