# Техническое задание
## Интеллектуальный помощник для Legion TD 2 (рабочее название: LT2 Copilot)

Версия: 2.0 · Дата: 17.07.2026 · Статус: актуальная

---

## 0. Резюме и ключевые архитектурные решения

Это pet-проект одного разработчика (вайбкодинг), без дедлайнов, для личного использования, с изучением Go и ML как основными задачами.

| Источник данных | Статус | Роль в архитектуре |
|---|---|---|
| Официальный API (`apiv2.legiontd2.com`, ключ через `developer.legiontd2.com`) | ✅ Подтверждено полной схемой (Swagger). Rate limit: 5 req/s, 100 burst, 1000/день | **Офлайн-контур**: обучение ML-модели, справочная база знаний, ground truth |
| HudApi игра (Coherent GT, через `patches/copilot-patcher.js`) | ✅ Используется в runtime | **Runtime-контур**: live-данные о состоянии матча (Mythium, Gold, Wave, Hand, Field, Mercs) |
| Overwolf GEP | ❌ Не используется | Исключено — приоритет на собственный пайплайн |
| OCR/CV | ❌ Удалено из проекта | Лишний компонент — все данные получаются через HudApi |
| Чтение памяти процесса | ❌ Запрещено | Красная линия, не рассматривается |

**Главный вывод:** проект — это Desktop-приложение (Go + Wails), которое получает live-данные матча через HudApi-патчер, применяет эвристические и ML-рекомендации, логирует матчи в SQLite. ML-модель тренируется на исторических данных официального API. Никакого OCR/CV/Overwolf.

---

## 1. Цели проекта

**Основная цель:** desktop-приложение под Windows, которое во время матча Legion TD 2 показывает пользователю рекомендации по расстановке юнитов, покупкам, продажам, найму Mercenaries и управлению Mythium.

**Цель для разработчика:** практическое изучение Go и ML через разработку промышленно-структурированного приложения.

**Не цели:** не автоматизация действий в игре, не бот, не чит, не публичный/коммерческий продукт.

---

## 2. Бизнес-требования

- BR1. Приложение бесплатно, для личного использования, без монетизации.
- BR2. Приложение не должно требовать модификации игрового клиента, инъекций в процесс или чтения памяти.
- BR3. Приложение не должно заметно ухудшать FPS игры.
- BR4. Проект должен допускать паузу/заморозку в любой момент без потери данных.
- BR5. Кодовая база должна быть документирована для возврата после перерыва.

---

## 3. Функциональные требования

### 3.1 Базовый функционал (Phase 1 — Desktop)
- FR1. Нативное Desktop-окно, показывающее в реальном времени: Mythium, Gold, Income, Wave, Timer, King HP.
- FR2. Отображение состава руки (hand юниты), поля (field units), очереди покупок.
- FR3. Эвристические рекомендации по тратам Mythium, контр-юнитам под волну.
- FR4. Логирование матчей в локальную SQLite.
- FR5. Индикатор «вижу игру уверенно / не вижу».

### 3.2 ML-функционал (Phase 2 — ML Training)
- FR6. Сбор и подготовка датасета из официального API (3000+ матчей).
- FR7. Выбор и обучение ML-модели для рекомендации расстановки юнитов.
- FR8. Выбор и обучение ML-модели для прогноза удержания волны.
- FR9. Интеграция обученной модели в Advisor.

### 3.3 Улучшения (Phase 3 — Refinement)
- FR10. Пост-матчевый отчёт.
- FR11. Рекомендации по найму Mercenaries.
- FR12. Текстовое объяснение рекомендаций ("почему").

### 3.4 Явно вне скоупа
- Автоматические клики/действия в игре.
- Любая функциональность, дающая скрытую от игрока информацию.

---

## 4. Нефункциональные требования

- NFR1. Задержка от игрового события до отображения рекомендации: ≤ 2 сек.
- NFR2. Нагрузка на CPU: не более 15–20% одного ядра в фоне.
- NFR3. Деградация при потере соединения с игрой, не краш.
- NFR4. Данные только локально (SQLite), без облачной синхронизации.
- NFR5. Никаких внешних запросов во время матча, кроме официального API (только чтение, не во время матча).

---

## 5. Ограничения

- C1. Разработчик обучается Go и ML — сроки индикативны.
- C2. Один разработчик, вайбкодинг (совместная разработка с AI-ассистентом).
- C3. Только Windows на первом этапе.
- C4. Игра патчится регулярно — данные HudApi могут меняться.

## 6. Архитектура

### 6.1 Общая схема

```
┌──────────────────────────────────────────────────────────────┐
│                   Windows Desktop Application                  │
│                   (Go + Wails Native Desktop)                  │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────────────┐  │
│  │  Wails UI    │  │    Advisor   │  │    SQLite Storage   │  │
│  │  (native     │◄─┤  (heuristic  │◄─┤  (matches, snaps,  │  │
│  │   window)    │  │   + ML)      │  │   recommendations)  │  │
│  └──────────────┘  └──────┬───────┘  └────────────────────┘  │
│                            │                                  │
│                     ┌──────▼───────┐                          │
│                     │  GameState   │                          │
│                     │  (Hub)       │                          │
│                     └──────▲───────┘                          │
└────────────────────────────┼──────────────────────────────────┘
                             │ WebSocket
              ┌──────────────┴──────────────┐
              │  Game Client                  │
              │  (patches/copilot-patcher.js) │
              │  — HudApi events → WS        │
              └──────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│  Офлайн ML-контур (Python, отдельно от runtime)               │
│  - cmd/dataset/ — сбор исторических данных из API v2          │
│  - Обучение модели рекомендаций (расстановка, удержание волны) │
│  - Экспорт модели для загрузки в Go Advisor                    │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 Компоненты

1. **Go Core Orchestrator** — главный процесс. Получает GameState из Hub, запускает Advisor, пишет в SQLite, отдаёт в UI.
2. **Wails UI** — нативное Desktop-окно с отображением состояния и рекомендаций.
3. **GameState Hub** — WebSocket hub, принимающий данные от игрового патчера и раздающий их подсистемам.
4. **SQLite** — runtime-логирование матчей и кэш справочных данных API.
5. **Офлайн ML-контур** — Python-скрипты сбора датасета и обучения моделей, не часть runtime.

### 6.3 Поток данных

1. HudApi игры → `copilot-patcher.js` → WebSocket → Hub (GameState)
2. Advisor (эвристики + ML) → рекомендации
3. Wails UI → отображение
4. SQLite → логирование

## 7. Требования к данным

- DR1. Скриншоты не используются — все данные из HudApi.
- DR2. История матчей из API кэшируется в SQLite.
- DR3. Схема БД версионируется через миграции.
- DR4. Локальные логи матчей — основной сигнал для ML-модели.

---

## 8. Требования к AI/ML

- ML1. Два независимых ML-контура: Perception больше нет, остаётся Advisor.
- ML2. Advisor стартует эвристическим, ML подключается по готовности модели.
- ML3. Все рекомендации сопровождаются текстовым объяснением.
- ML4. Инференс — только локально, без внешних AI API.
- ML5. Датасет: 3000+ матчей из API v2 (buildPerWave, leaksPerWave, netWorthPerWave).
- ML6. Модель: исследовать варианты (XGBoost, LightGBM, небольшая NN). Определить метрики качества.
- ML7. Training pipeline: отдельные Python-скрипты, экспорт модели в ONNX или Go-совместимый формат.

---

## 9. Требования к безопасности

- SEC1. Запрет на чтение/инъекцию в процесс игры. HudApi через Coherent GT — штатный механизм игры, не инъекция. `copilot-patcher.js` работает в рамках разрешённого API.
- SEC2. Приложение не модифицирует файлы игры.
- SEC3. Данные не покидают локальную машину.
- SEC4. EULA/ToS прочитан, явных блокеров не выявлено.
- SEC5. Никаких сетевых функций, публикации данных.

---

## 10. Требования к UI/UX

- UX1. Нативное Desktop-окно (Wails), отдельное от игры.
- UX2. Крупные цифры (Mythium, Gold, Income, HP, Wave, Timer).
- UX3. Список рекомендаций с приоритетом и текстовым объяснением.
- UX4. RU/EN переключаемый язык интерфейса.
- UX5. История волн текущего матча.

## 11. Требования к логированию

- LOG1. Структурированное логирование (Go: `slog`).
- LOG2. Отдельный лог игровых состояний (в SQLite).
- LOG3. Debug-режим для отладки.

## 13. Риски

| ID | Риск | Влияние | Митигирование |
|---|---|---|---|
| Р1 | HudApi изменится после патча игры | Среднее | Мониторинг структуры данных, fallback на базовые эвристики |
| Р2 | Недостаточно данных для ML | Среднее | Advisor остаётся эвристическим, ML — опционально |
| Р3 | copilot-patcher.js перестанет работать | Высокое | Ручная фиксация версии игры, обновление патчера |
| Р4 | Разработчик — новичок в Go/ML | Среднее | Чистая архитектура, ADR, unit-тесты |
| Р5 | Качество ML-модели недостаточно | Среднее | Итеративный подход, метрики качества |

## 14. Критерии готовности

**Phase 1 (Desktop):**
- Приложение стабильно отображает Mythium, Gold, Income, Wave, Timer, King HP.
- Рекомендации Advisor отображаются.
- SQLite-логирование работает.
- UI — нативное Wails-окно.

**Phase 2 (ML Training):**
- ML-модель выбрана и обучена на датасете.
- Модель интегрирована в Advisor.
- Метрики качества зафиксированы.

**Phase 3 (Refinement):**
- Пост-матчевый отчёт.
- Рекомендации с текстовым объяснением.
- Ручное тестирование пройдено.
