# AGENTS.md — LT2 Copilot

Этот файл подключён в `opencode.json` как `instructions` и читается всеми агентами
в дополнение к их собственным системным промптам.

## Что это за проект

Десктопный ассистент для Legion TD 2: real-time рекомендации во время матча.
Соло pet-проект, вайб-кодинг, без коммерческих целей. Полное ТЗ — в
`TZ_LegionTD2_Assistant.md` в корне репозитория, это источник истины при любом
конфликте между тем, что "уже написано в коде", и тем, что задокументировано как
намерение.

## Прежде чем писать код — прочитай красные линии

Скилл `ltd2-constraints` (`.opencode/skills/ltd2-constraints/SKILL.md`) содержит
неизменяемые ограничения проекта — в первую очередь запрет на любую инъекцию/
модификацию процесса игры. Если задача в любой мере касается того, как приложение
получает данные о состоянии матча — загрузи этот скилл прежде, чем предлагать решение.

## Двухслойная система скиллов

Проект использует **два слоя** скиллов, не смешивая их:

1. **Фундамент** — пакет [addyosmani/agent-skills](https://github.com/addyosmani/agent-skills)
   (24 продакшн-скилла: spec-driven-development, test-driven-development,
   code-review-and-quality, security-and-hardening, api-and-interface-design,
   git-workflow-and-versioning, ci-cd-and-automation, documentation-and-adrs,
   observability-and-instrumentation и др.). Это общеинженерная дисциплина,
   не завязанная на Legion TD 2 — она должна применяться по умолчанию на ЛЮБОМ
   проекте, не только на этом.
2. **Проектная специфика** — скиллы `ltd2-*` в `.opencode/skills/` этого репозитория.
   Они не дублируют фундамент, а добавляют то, что специфично именно для LT2 Copilot
   (красные линии из ТЗ, формат API v2, границу Go/Python).

**Правило приоритета:** при конфликте между общей инженерной практикой из фундамента
и проектным скиллом — проектный скилл (`ltd2-constraints` в первую очередь) имеет
приоритет, потому что он кодирует юридические/архитектурные ограничения конкретно
этого проекта, а не общие best practices.

**Дефолтное применение фундамента** (агенты должны делать это без явного запроса
пользователя, как и задумано в дизайне пакета):
- Начало новой фичи → `spec-driven-development`, затем `planning-and-task-breakdown`.
- Реализация логики → `test-driven-development` (red-green-refactor), не "напишу тесты потом".
- Любой коммит → `git-workflow-and-versioning` (atomic commits, ~100 строк на изменение).
- Перед мержем → `code-review-and-quality` + `security-and-hardening` (последний — обязателен
  для всего, что трогает `internal/http`, `internal/ws`, `internal/api` — там либо
  пользовательский ввод, либо внешние данные).
- Проектирование любого контракта (gRPC/REST между Go и Python, эндпоинты API) →
  `api-and-interface-design`.
- Архитектурное решение → `documentation-and-adrs` (создать ADR), не просто "написал и забыл".

## Какой агент для какой задачи

| Задача | Агент |
|---|---|---|
| Go: advisor, storage, http/ws, api-клиент | `@go-core` (primary) |
| ML: датасет, обучение моделей, интеграция | `@ml-dataset` (primary) |
| Проверить архитектурное решение на соответствие ТЗ | `@architecture-guard` (subagent, read-only) |
| Ревью качества кода (не архитектуры) | `@code-reviewer` (subagent, read-only) |

Дополнительно из фундамента доступны персоны `@code-reviewer` (пакета addyosmani,
пятиосевое ревью), `@test-engineer` и `@security-auditor`.

`@architecture-guard` стоит вызывать явно перед любыми архитектурными изменениями.

## Известные архитектурные расхождения

1. Данные о юнитах дублируются: `internal/unitdata/units.go` (статика) и
   `units_reference` (кэш API) — при правке одного проверяй, не разошёлся ли другой.
2. HTTP-сервер + WebSocket + браузерный UI существуют как временное решение —
   этап 1.1 Phase 1: замена на Wails Desktop UI.
3. ML-модель не выбрана и не обучена — задача Phase 2.
