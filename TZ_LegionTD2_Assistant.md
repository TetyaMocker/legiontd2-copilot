# Техническое задание
## Интеллектуальный помощник для Legion TD 2 (рабочее название: LT2 Copilot)

Версия: 1.0 · Дата: 16.07.2026 · Статус: черновик для старта разработки

---

## 0. Резюме и ключевые архитектурные решения

Это pet-проект одного разработчика (вайбкодинг), без дедлайнов, для личного использования, с изучением Go как основной задачей. На основе исследования принято решение об архитектуре данных, которое существенно отличается от изначального плана:

| Источник (по приоритету из брифа) | Статус после проверки | Роль в архитектуре |
|---|---|---|
| Официальный API (`apiv2.legiontd2.com`, ключ через `developer.legiontd2.com`) | ✅ **Подтверждено полной схемой (Swagger).** Даёт: профили игроков, историю матчей `/games` (не старше 1 года, старые — архив `.bson` на 6 частей), детальную статистику юнитов по патчам `/units`, справочник волн/легионов/спеллов/абилок/исследований `/info`. Rate limit: 5 req/s, 100 burst, 1000/день (можно повысить через Discord) | **Офлайн-контур**: обучение модели, справочная база знаний, ground truth. См. п.9.1 ниже — данные оказались гораздо богаче ожидаемого |
| Replay-файлы | ❌ Не подтверждено наличие полноценных реплеев (по отзывам игроков — есть только спектейт) | Исключено из MVP |
| Игровые логи (client log) | ❌ **Проверено пользователем — ничего похожего на структурированные логи не найдено** | Исключено из архитектуры |
| Overwolf GEP (Game Events Provider) | ❓ Ещё не проверено на практике (единственный оставшийся открытый пункт Phase 0) | Проверяется отдельно, не блокирует старт разработки — MVP проектируется без него, с возможностью подключить как upgrade |
| OCR | Основной live-компонент | **Runtime-контур**: значения (HP, Mythium, таймеры) |
| Computer Vision | Возможно, менее критичен на MVP, чем считалось (см. ниже) | **Runtime-контур**: расположение юнитов на поле боя — важен для Фазы 2, не для MVP |
| Чтение памяти процесса | Технически возможно (нет kernel-level anti-cheat — есть открытые трейнеры), юридически нежелательно | **Исключено из проекта.** Красная линия, не «последний вариант» |
| EULA/ToS | Прочитан пользователем, явных блокеров не выявлено (личное некоммерческое использование) | Риск Р1 понижен, но не закрыт полностью — см. раздел 18 |

**Главный вывод:** проект — это не «получить данные», а «получить данные достаточно точно и достаточно быстро через runtime-конвейер OCR/CV + опциональный Overwolf-канал», плюс отдельный офлайн ML-контур на официальном API. Это меняет распределение сложности: Python/CV-часть будет технически тяжелее, чем предполагалось, Go остаётся ядром оркестрации и продукта.

**Юридический статус:** текст EULA/ToS Legion TD 2 не найден и не прочитан целиком — это открытый риск (см. раздел 18, п. Р1). FairPlay — поведенческая система (детектит ливеров/AFK/токсичность по игровым событиям и репортам), она не анализирует сторонние процессы. Kernel-level anti-cheat (EAC/BattlEye/VAC) не обнаружен.

## 1. Цели проекта

**Основная цель:** desktop-приложение под Windows, которое во время (или сразу после) матча Legion TD 2 показывает пользователю рекомендации по расстановке юнитов, покупкам, продажам, найму Mercenaries и управлению Mythium, основанные на анализе текущего состояния игры и статистике, недоступной для мгновенного расчёта в уме.

**Цель для разработчика:** практическое изучение Go через разработку промышленно-структурированного приложения; получение опыта в ML/CV на прикладной задаче.

**Не цели:** не автоматизация действий в игре, не бот, не чит, не публичный/коммерческий продукт (на данном этапе).

---

## 2. Бизнес-требования

- BR1. Приложение бесплатно, для личного использования, без монетизации.
- BR2. Приложение не должно требовать модификации игрового клиента, инъекций в процесс или чтения памяти.
- BR3. Приложение может использовать GPU/CPU пользователя, но не должно заметно ухудшать FPS игры.
- BR4. Проект должен допускать паузу/заморозку в любой момент без потери данных (личный проект без дедлайнов).
- BR5. Кодовая база должна быть достаточно документирована, чтобы автор (не имеющий опыта в Go/ML) мог вернуться к проекту после перерыва и понять архитектуру.

---

## 3. Функциональные требования

### 3.1 MVP (Фаза 1)
- FR1. Оверлей/отдельное окно, показывающее в реальном времени: Mythium, доход, King HP (свой и союзника, если доступно).
- FR2. Распознавание волны (номер, таймер до следующей волны).
- FR3. Базовая рекомендация: сколько Mythium стоит потратить/оставить на этой волне (эвристика, не ML, на старте).
- FR4. Логирование распознанных состояний матча в локальную SQLite-базу для последующего анализа и обучения.
- FR5. Индикатор состояния распознавания («вижу поле уверенно / не уверен / не вижу») — обязателен по вашему пункту 3.6 ("подавать признаки жизни").

### 3.2 Фаза 2
- FR6. Распознавание состава юнитов на своей линии (тип, количество, примерная позиция).
- FR7. Рекомендация расстановки для новой волны на основе истории и типа входящей волны.
- FR8. Рекомендации по покупке/продаже юнитов с текстовым объяснением ("почему").
- FR9. Учёт линии союзника (частично, по мере доступности данных).

### 3.3 Фаза 3
- FR10. Прогноз вероятности удержания следующей волны (ML-модель, обученная на исторических данных + локальной статистике игрока).
- FR11. Рекомендации по найму Mercenaries и таймингу отправки.
- FR12. Учёт армий обоих противников (если видимы через спектейт/дополнительный источник).
- FR13. Пост-матчевый отчёт: что пошло не так, где были неоптимальные решения (на основе лога FR4).

### 3.4 Явно вне скоупа
- Автоматические клики/действия в игре.
- Обход matchmaking, ranked-ограничений и т.п.
- Любая функциональность, дающая информацию, скрытую от игрока по игровому дизайну (например, содержимое закрытых Reroll-слотов оппонента), если она не наблюдаема визуально.

## 4. Нефункциональные требования

- NFR1. Задержка от игрового события до отображения рекомендации: цель ≤ 2 сек для MVP, ≤ 500 мс в Фазе 3 (уточняется в Phase 0 по факту замеров).
- NFR2. Нагрузка на CPU от приложения-помощника: не более 15–20% одного ядра в фоне, всплески при инференсе — не более 1 сек.
- NFR3. Приложение должно штатно работать при потере распознавания (деградация, а не краш).
- NFR4. Данные пользователя (скриншоты, логи матчей) хранятся только локально (SQLite), без облачной синхронизации в MVP.
- NFR5. Приложение самостоятельно ничего не публикует и не отправляет во внешние сервисы, кроме обращений к официальному API Legion TD 2 (только для чтения истории/статистики, не во время матча).

## 5. Ограничения

- C1. Разработчик обучается Go в процессе — сроки индикативны, приоритет — учебная ценность, а не скорость.
- C2. Один разработчик, вайбкодинг (совместная разработка с AI-ассистентом) — архитектура должна быть простой для навигации и хорошо закомментирована/задокументирована.
- C3. Только Windows на первом этапе (согласно 5.1).
- C4. Игра патчится регулярно — UI-координаты и, возможно, визуальные элементы могут ломать CV/OCR разметку. Разметка должна быть конфигурируемой (не хардкод), с версионированием под патчи.
- C5. Нет подтверждённого доступа к replay/логам на момент старта — Phase 0 обязана это прояснить до проектирования CV-пайплайна.

## 6. Допущения

- A1. Игра не использует kernel-level anti-cheat (подтверждено косвенно: FairPlay — поведенческая система; существуют публичные трейнеры).
- A2. Overwolf, если подтвердит поддержку live game events для Legion TD 2, будет приоритетным источником данных над чистым CV (точнее, дешевле, безопаснее).
- A3. Официальный API v2 доступен без сложной авторизации для чтения публичных данных (профили, история) — уточняется в Phase 0 по документации developer-портала.
- A4. Разрешение экрана и UI-масштаб пользователя фиксированы или ограничены небольшим набором поддерживаемых конфигураций на MVP (полная инвариантность к разрешению — Фаза 2+).

## 7. Пользовательские сценарии

**US1.** Пользователь запускает Legion TD 2, затем запускает LT2 Copilot. Приложение обнаруживает запущенный процесс игры и активируется автоматически.

**US2.** Во время волны построения пользователь смотрит на отдельное окно рядом с игрой (второй монитор или оконный режим игры), видит текущие Mythium/доход/рекомендацию и решает, что купить.

**US3.** После волны приложение обновляет статистику (удержана/не удержана, потери) и логирует это для последующего анализа.

**US4.** После завершения матча пользователь открывает пост-матчевый отчёт с разбором ключевых решений (Фаза 3).

**US5.** Если распознавание сбоит (например, после патча игры сломалась разметка UI), пользователь видит явный индикатор «низкая уверенность» вместо тихого некорректного совета.

## 8. Архитектурные требования

### 8.1 Общая схема (высокоуровнево)

```
┌─────────────────────────────────────────────────────────────┐
│                      Windows Desktop App                      │
│                    (Go, Wails, UI-слой)                       │
└───────────────┬─────────────────────────────┬─────────────────┘
                │ gRPC                         │ gRPC
┌───────────────▼───────────┐   ┌─────────────▼──────────────────┐
│   Go: Core Orchestrator     │   │  Python: Perception Service      │
│  - оркестрация pipeline     │   │  - захват экрана                 │
│  - бизнес-правила/рекоменд. │◄──┤  - OCR (Mythium/HP/таймер)       │
│  - хранение (SQLite)        │   │  - CV (юниты, позиции)           │
│  - Overwolf-коннектор (если │   │  - ONNX Runtime инференс          │
│    подтверждён в Phase 0)   │   └──────────────────────────────────┘
└───────────────┬─────────────┘
                │
     ┌──────────▼───────────┐        ┌─────────────────────────┐
     │  SQLite (runtime,     │        │  Офлайн ML-контур         │
     │  локальная история)   │        │  (Python, отдельно от      │
     └───────────────────────┘        │  runtime): обучение на      │
                                       │  данных официального API    │
                                       └─────────────────────────┘
```

### 8.2 Компоненты

1. **Go Core Orchestrator** — координирует запросы к Perception Service, применяет бизнес-логику/эвристики/ML-инференс рекомендаций, пишет в SQLite, отдаёт данные в UI.
2. **Python Perception Service** — отдельный процесс, gRPC-сервер. Инкапсулирует весь CV/OCR/ONNX. Go никогда не работает с изображениями напрямую.
3. **UI (Wails, Go+веб-фронтенд)** — отдельное окно, как решили в п. 5.2. Overlay-режим — отложен на Фазу 2+ (технически сложнее, требует click-through/always-on-top, что не критично для MVP).
4. **Data/Training pipeline (офлайн, отдельно от runtime)** — Python-скрипты, обращающиеся к официальному API v2 и к локально накопленным логам матчей (FR4), для тренировки моделей рекомендаций. Не часть runtime-приложения, запускается вручную/по расписанию разработчиком.
5. **Overwolf-коннектор (опционально, по итогам Phase 0)** — если Overwolf отдаёт полезные live-события, отдельный модуль на Go/JS-мостике, который заменяет часть OCR-функций более надёжным источником.

### 8.3 Обоснование выбора Go+Python-разделения

Один процесс на Go, который вызывал бы Python через FFI/subprocess, создал бы хрупкую связь и усложнил отладку. Разделение через gRPC даёт: независимое развитие каждого сервиса, возможность профилировать/тестировать Perception Service отдельно, и чёткую границу ответственности, что важно для вайбкодинга — легче объяснить AI-ассистенту «поправь только сервис X».

## 8.4 Находка, меняющая приоритеты MVP: `buildPerWave`

Разбор реальной Swagger-схемы показал, что объект матча (`/games`, `/games/byId/{id}` с `includeDetails=true`) в `playersData` содержит по каждому игроку массивы **на каждую волну**, а не только на игру целиком:

- `buildPerWave` — список юнитов **с координатами расстановки** (`"fire_elemental_unit_id:0.5|6"`) на каждой волне;
- `netWorthPerWave`, `workersPerWave`, `incomePerWave` — экономика по волнам;
- `mercenariesSentPerWave` / `mercenariesReceivedPerWave` — что и когда отправлено;
- `leaksPerWave` — какие юниты противника прошли (протекли) через оборону — это и есть сигнал "волна не удержана";
- `kingUpgradesPerWave` / `opponentKingUpgradesPerWave` — апгрейды короля по волнам;
- `leftKingPercentHp` / `rightKingPercentHp` на уровне матча — % HP короля по волнам для обеих сторон.

**Почему это важно:** это означает, что **расстановку юнитов с координатами и результат (протекло/не протекло) можно получить из истории тысяч реальных матчей без всякого CV**, просто вызывая `/games?includeDetails=true` за прошедший год (плюс исторический архив 2018–2021 через `.bson`-дамп). Это готовый датасет для обучения:
- рекомендатора расстановки (позиция → волна → удержана/протекла);
- прогноза вероятности удержания волны (используя net worth, состав, историю).

**Ограничение:** `unitClass` в схеме юнитов подтверждает деление на Creature/Fighter/King/Mercenary/Worker — это разработчик API специально документировал для внешних инструментов, значит такое использование ожидаемо и не выглядит как обход правил.

Это меняет распределение усилий Фазы 2: вместо того чтобы сразу разрабатывать CV-детекцию юнитов на поле, для **первой версии рекомендателя расстановки** достаточно офлайн-модели на исторических данных API. Live CV-детекция позиций становится задачей персонализации ("а что у МЕНЯ на поле прямо сейчас"), а не единственным источником обучающих данных.

## 9. Требования к данным

- DR1. Все скриншоты, используемые для распознавания в runtime, не сохраняются на диск по умолчанию (только в памяти), кроме явного debug-режима (для разметки датасета).
- DR2. Датасет для обучения CV-моделей размечается вручную самим пользователем (согласно ответу 3.5) — необходим встроенный инструмент разметки (простой, отдельный скрипт/mini-UI) уже в Phase 0/1.
- DR3. История матчей из официального API кэшируется локально в SQLite/Postgres (по объёму данных) с TTL-обновлением, чтобы не дёргать API избыточно.
- DR4. Схема БД версионируется через миграции (см. xlsx/скорее — обычные Go-миграции: `golang-migrate` или аналог).
- DR5. Локальные логи распознанных состояний матчей (FR4) — это основной обучающий сигнал для «своей» модели (в отличие от чужой статистики из API), и должны храниться в структурированном виде, пригодном для последующего экспорта в training pipeline.

## 10. Требования к AI/ML

- AI1. Архитектурно: два независимых ML-контура — (а) **Perception** (CV/OCR: распознать, что на экране) и (б) **Advisor** (рекомендации: что делать при данном состоянии). Их нельзя смешивать в одну модель — иначе невозможно отлаживать отдельно.
- AI2. Advisor стартует как **эвристический движок** (правила на основе игровой логики, без ML) — это осознанное решение: ML без достаточного объёма данных на старте даст непредсказуемые и необъяснимые рекомендации. ML подключается в Фазе 3, когда накоплены логи (DR5) и/или данные API.
- AI3. Все рекомендации сопровождаются текстовым объяснением (FR8) — важно и для доверия пользователя, и для отладки самим разработчиком.
- AI4. Инференс — только локально (по вашему ответу 6.3): ONNX Runtime для CV-моделей, без внешних вызовов к облачным AI API в runtime-контуре.
- AI5. Метрика качества Perception — не абстрактная accuracy, а операционная: доля кадров, где ключевые величины (Mythium, HP, номер волны) распознаны с ошибкой, не влияющей на рекомендацию.

## 11. Требования к безопасности

- SEC1. **Запрет на чтение/инъекцию в процесс игры** — абсолютное ограничение архитектуры, не список приоритетов.
- SEC2. Приложение не модифицирует файлы игры, не патчит клиент.
- SEC3. Скриншоты и данные не покидают локальную машину, за исключением опциональных вызовов к официальному публичному API (без передачи персональных данных сверх того, что публично в профиле).
- SEC4. Перед первым релизом даже "для себя" — обязательно прочитать текст EULA/ToS Legion TD 2 целиком и оценить, попадает ли приложение под определения "third-party tool"/"unauthorized software" (открытый риск Р1, раздел 18).
- SEC5. Приложение не должно взаимодействовать с другими игроками (никаких сетевых функций, публикации данных матчей и т.п. в MVP).

## 12. Требования к UI/UX

- UX1. Отдельное окно (не overlay) в MVP, размещаемое пользователем самостоятельно (второй монитор/оконный режим).
- UX2. Минималистичный, читаемый на бегу интерфейс: крупные цифры (Mythium, HP, таймер), явный индикатор уверенности распознавания (US5/FR5), краткие текстовые рекомендации.
- UX3. Возможность просмотра истории волн текущего матча (что было куплено, что рекомендовано, что реально произошло) — полезно и для игрока, и для отладки Perception/Advisor.
- UX4. Язык интерфейса: RU/EN (переключаемый), согласно ответу 10.2.

## 13. Требования к производительности

- PERF1. Perception Service: частота захвата экрана и инференса — не выше необходимой (например, 1–2 кадра/сек для OCR-величин, событийно/по требованию для CV юнитов), чтобы не конкурировать с игрой за GPU/CPU.
- PERF2. Целевое железо — ПК пользователя (i5-13400F, RTX 4070 Ti SUPER, 32 ГБ ОЗУ, см. скриншот) — производительность достаточна для локального ONNX-инференса без проблем; но код не должен предполагать наличие GPU как обязательное условие (fallback на CPU для воспроизводимости на других машинах, если проект когда-либо станет публичным).
- PERF3. Замеры латентности (NFR1) фиксируются как часть Phase 0/1 — до этого цифры в NFR1 являются ориентировочными, не контрактными.

## 14. Требования к логированию

- LOG1. Структурированное логирование (Go: `slog` или `zap`; Python: `structlog`/стандартный `logging` в JSON) с уровнями DEBUG/INFO/WARN/ERROR.
- LOG2. Отдельный лог распознанных игровых состояний (FR4/DR5) — не смешивать с техническим логом приложения.
- LOG3. Debug-режим, сохраняющий сырые скриншоты с разметкой (bounding boxes) для ручной проверки качества CV — включается явно, не по умолчанию (см. DR1).

## 15. Требования к мониторингу

Для pet-проекта полноценный мониторинг (Prometheus/Grafana) избыточен на MVP, но закладываем совместимость:

- MON1. Perception Service экспортирует базовые метрики (латентность инференса, confidence score) через простой HTTP `/metrics` эндпоинт (Prometheus-формат) — можно не разворачивать стек мониторинга сразу, но формат делает это тривиальным позже.
- MON2. В UI отображается упрощённая версия этих метрик для пользователя (UX2/FR5).

## 16. Требования к тестированию

- TEST1. Go-сервисы: unit-тесты на бизнес-логику Advisor (эвристики/правила) — это тестируется легче всего и с наибольшей пользой, приоритет №1.
- TEST2. Python Perception: тесты на фиксированном наборе размеченных скриншотов (regression-тест на точность OCR/CV при обновлении моделей).
- TEST3. Golden-датасет (набор скриншотов с известным правильным ответом) обязателен до начала Фазы 2 — иначе невозможно объективно оценивать прогресс CV.
- TEST4. Интеграционный тест gRPC-контракта между Go и Python (schema compatibility).

## 17. CI/CD и DevOps

- DEVOPS1. Docker используется для Perception Service (Python + зависимости CV/ONNX) — воспроизводимость окружения важнее для этой части, где больше нативных зависимостей (OpenCV и т.п.).
- DEVOPS2. Go-сервис и Wails-приложение собираются нативно под Windows (Docker не даёт особой пользы для desktop GUI).
- DEVOPS3. GitHub Actions (раз проект будет вестись в git, даже приватном) — линтеры (golangci-lint, ruff/black для Python), запуск unit-тестов на пуш. CI/CD в полном смысле (авто-деплой) не нужен для локального desktop-приложения.
- DEVOPS4. Версионирование релизов — простое semver-тегирование, без автообновления в MVP.

## 19. Критерии готовности

**Готовность MVP (Фаза 1):**
- Приложение стабильно (без крашей за сессию) отображает Mythium, доход, номер волны и таймер с задержкой ≤ 2 сек и точностью, subъективно достаточной для принятия решений в 9/10 волн.
- Индикатор уверенности распознавания работает и корректно понижается при деградации.
- Базовые эвристические рекомендации по тратам Mythium отображаются.
- Логирование матчей в SQLite работает без потери данных.

**Готовность Фазы 2:** распознавание юнитов на своей линии с приемлемой точностью (порог определяется golden-датасетом, TEST3), первые персонализированные рекомендации по составу.

**Готовность Фазы 3:** ML-модель прогноза удержания волны обучена и валидирована на отложенной выборке, пост-матчевый отчёт функционирует.

0.6. Сделать 20–30 скриншотов разных состояний игры для оценки сложности OCR (для Phase 1.2).

### Phase 1 — MVP (Фаза 1 функциональности, раздел 19)
1.1. Каркас проекта: репозиторий, структура Go-модулей, Python Perception Service со "пустым" gRPC-контрактом, docker-compose для дев-окружения.
1.2. Клиент для официального API v2 (Go): получение ключа, кэширование справочников `/units`, `/info` в SQLite.
1.3. OCR для Mythium/HP/таймера/номера волны (простейший вариант — Tesseract или EasyOCR, оценить по скриншотам из 0.6).
1.4. SQLite-схема и запись логов текущих матчей пользователя (runtime).
1.5. Wails-окно с базовым UI (UX1-UX3).
1.6. Эвристический Advisor v0 (простые правила расхода Mythium, опираясь на справочник юнитов из 1.2).

### Phase 2 — Офлайн-датасет и рекомендации по составу (порядок скорректирован после находки 8.4)
2.1. **Приоритет 1:** сбор исторического датасета через `/games?includeDetails=true` (буфер за последний год) + разбор архива 2018–2021 (`.bson`) — сразу даёт `buildPerWave`, `leaksPerWave`, `netWorthPerWave` без CV.
2.2. Офлайн-модель рекомендателя расстановки/прогноза удержания волны на этом датасете (Python, отдельно от runtime).
2.3. Только после 2.1–2.2, если нужна персонализация под текущую партию: разметка датасета позиций юнитов на скриншотах + CV-модель детекции юнитов (ONNX), интеграция в Perception Service.
2.4. Advisor v1 — объединяет офлайн-модель (2.2) и live-данные (OCR + опционально CV) в рекомендации по составу.

### Phase 3 — ML Advisor и прогнозирование
3.1. Офлайн pipeline на данных официального API + накопленных локальных логах.
3.2. Модель прогноза удержания волны.
3.3. Пост-матчевый отчёт.
